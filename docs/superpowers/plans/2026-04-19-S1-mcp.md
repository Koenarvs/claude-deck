# S1 — MCP Server

**Burst:** Support | **Depends on:** F0 merged | **Branch:** `feat/S1-mcp`

## Goal
Standalone MCP server in `mcp/` subpackage that exposes claude-deck operations as tools any `claude` session can call.

## Spec references
- §11 MCP server — full contract and integration pattern
- §14.4 S1

## Scope
- Create: `mcp/package.json`
- Create: `mcp/tsconfig.json`
- Create: `mcp/src/index.ts` — MCP server boot, tool registry
- Create: `mcp/src/tools/*.ts` — one file per tool
- Create: `mcp/src/api-client.ts` — typed fetch wrapper to `http://127.0.0.1:4100/api/*`
- Create: `mcp/README.md` — install + config instructions

## Contracts consumed
- `src/shared/types.ts` (via relative import `../../src/shared/types.ts` since `mcp/` is a sibling — tsconfig path alias or relative import both fine)
- Backend API from B3-B6 at `http://127.0.0.1:4100`

## Tools to expose (spec §11)
- `list_goals(status?, tag?)` → GET /api/goals
- `get_goal(id)` → GET /api/goals/:id
- `create_goal(title, cwd, model?, initialPrompt?, tags?)` → POST /api/goals
- `send_message(goal_id, prompt)` → POST /api/goals/:goal_id/messages
- `list_sessions(origin?, active?)` → GET /api/sessions
- `get_session_messages(session_id)` → GET /api/sessions/:session_id/messages
- `schedule_task(name, cron_expr, goal_template)` → POST /api/scheduled-tasks

## Recommended task order
1. Scaffold `mcp/` package with its own `package.json` and `tsconfig.json`. Scripts: `build` (tsc), `start` (node dist/index.js), `dev` (tsx src/index.ts), `typecheck`.
2. TDD `api-client.ts`: thin typed wrapper around fetch for each endpoint; returns parsed domain types via zod validation.
3. TDD each tool file (e.g., `list_goals.ts`): takes MCP input, calls api-client, returns MCP output. Input/output schemas via zod.
4. `index.ts`: creates MCP server on stdio transport, registers all tools, handles tool-call dispatch.
5. Test locally: `mcp/dist/index.js` runs; MCP inspector can connect and list tools; `create_goal` round-trip produces a goal in SQLite.

## Integration pattern (spec §11)
- MCP tools call the HTTP API, NOT SQLite directly.
- This ensures MCP-driven writes flow through the same validation + broadcast as UI writes.
- An MCP `create_goal` results in a `goal:created` WS broadcast → live UI update.
